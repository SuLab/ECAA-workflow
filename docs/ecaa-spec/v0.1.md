# Evidence-Carrying Analysis Artifact (ECAA) — Specification v0.1

**Status:** Draft. PAR-26-040 Aim 1 deliverable D7.
**Date:** 2026-05-18.
**Profile IRI:** `https://w3id.org/ecaa/v0.1` (see §11).
**Media type:** `application/vnd.ecaa+zip` (see §11).

## Abstract

The Evidence-Carrying Analysis Artifact (ECAA) is a portable, machine-checkable conformance contract for AI-generated bioinformatic analyses. An ECAA package is a typed object `P = (I, D, E, V, C, Q, F, A)` over a Workflow Run RO-Crate (WRROC) v0.5 Tier-3 substrate, binding eight sub-graphs (Intent, Decision, Execution, Evidence, Claim, Equivalence, Failure, Audit-proof) under four validity-preserving operations (compose, amend, re-execute, ablate) and six machine-checkable audit-proof invariants. This document is the v0.1 normative specification. Implementation-independence is achieved by a conformance suite (D9, separate deliverable) that admits any implementation passing the five conformance bars defined in §8 without requiring reuse of the reference implementation's source.

## 1. Scope

This specification defines:

1. The typed object model (8 sub-graphs with closed node-type and edge-predicate sets) — §4-5.
2. Six audit-proof invariants and their default verdict semantics — §6 (with normative predicates in `invariants.md`).
3. Four validity-preserving operations — §7 (with normative contracts in `operations.md`).
4. The five-item conformance bar — §8.
5. A version negotiation protocol — §9.
6. Security considerations — §10.
7. IANA + registry considerations — §11.

This specification does NOT define:

- A second substrate binding beyond WRROC v0.5 Tier-3. (A BCO/IEEE-2791 binding is reserved for v0.2.)
- Domain-specific profiles (e.g., "ECAA Bulk-RNAseq Profile"). Reserved for v0.2+.
- Open extension registries for sub-graph node types, edge predicates, BlockerKind variants, Q-class names, invariants, or verdict states. v0.1 is fully closed; additive changes land in v0.2 via SemVer minor bumps.
- Hard-block invariant policy as a normative requirement. The default operational policy is warn-only; hard-block is informative-only.

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**, **SHALL NOT**, **SHOULD**, **SHOULD NOT**, **RECOMMENDED**, **MAY**, and **OPTIONAL** in this document are to be interpreted as described in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) when they appear in all capitals.

## 3. Substrate

ECAA v0.1 packages are WRROC v0.5 Tier-3 RO-Crates. The descriptor `ro-crate-metadata.json` is the package's authoritative entry point.

### 3.1 Required `conformsTo` profile IRIs

The package's root dataset MUST declare exactly these six profile IRIs in its `conformsTo` array (additional implementation-specific IRIs are PERMITTED but MUST NOT replace any of these six):

```
https://w3id.org/ro/crate/1.1
https://w3id.org/workflowhub/workflow-ro-crate/1.0
https://w3id.org/ro/wfrun/process/0.5
https://w3id.org/ro/wfrun/workflow/0.5
https://w3id.org/ro/wfrun/provenance/0.5
https://w3id.org/ecaa/v0.1
```

The first IRI (RO-Crate 1.1) establishes the base format. The next four (WRROC v0.5 Process/Workflow/Provenance) establish Tier-3 conformance. The last (this specification's profile IRI) identifies the package as ECAA-v0.1-conformant.

### 3.2 Four post-checks

Beyond `runcrate validate ≥ 0.5.0`, ECAA-v0.1-conformant readers MUST perform these four post-checks:

1. All six profile IRIs in §3.1 are present in `conformsTo`.
2. At least one `@graph` entity has `@type` containing `wfprov:ParameterConnection`.
3. At least one `@graph` entity has `@type` containing `p-plan:Plan`.
4. Every required `runtime/*.jsonl` and `runtime/*.json` listed in §3.4 is referenced as a `CreativeWork` dataset member in the `@graph`.

Failure of any post-check is a `Fail` verdict for the `substrate_validity` audit-proof invariant (Invariant 6).

### 3.3 Package boundary

An ECAA v0.1 package is a directory tree (or `.zip` archive thereof) with the following root contents:

- `ro-crate-metadata.json` — REQUIRED. The WRROC descriptor.
- `runtime/` — REQUIRED. Subdirectory holding the eight sub-graph sidecars (§3.4).
- Data files referenced by the descriptor (workflow inputs, outputs, figures, etc.) at any paths declared in the descriptor.

Readers MUST tolerate additional files at the root or under `runtime/` (forward-compatibility per §9.4).

### 3.4 Required sidecars

| Sub-graph | Required sidecar | Format |
|---|---|---|
| I — Intent | `runtime/intake-conversation.jsonl` | JSON Lines |
| D — Decision | `runtime/decisions.jsonl` | JSON Lines |
| E — Execution | `runtime/validation-reports.jsonl` (typed shadow; the WRROC `@graph` is authoritative — see §5.3) | JSON Lines |
| V — Evidence | `runtime/proofs.jsonl` | JSON Lines |
| C — Claim | `runtime/claim-verification.json` | JSON |
| Q — Equivalence | `runtime/verifier-decisions.jsonl` | JSON Lines |
| F — Failure | `runtime/assumptions.jsonl` | JSON Lines |
| A — Audit-proof | `runtime/audit-proof-report.json` | JSON |

A package is v0.1-conformant only if all eight sidecars are present. A package emitted with one or more `ECAA_ABLATE_*` flags (see §7.4) is by definition non-conformant — the suppressed sidecar is absent.

### 3.5 Media type

The IANA media type for an ECAA package is `application/vnd.ecaa+zip` (zipped form) with `application/vnd.ecaa+directory` as an informational alias for the unpacked filesystem form. See §11.

## 4. The ECAA typed object

An ECAA v0.1 package is a typed tuple

```
P = (I, D, E, V, C, Q, F, A)
```

where each component is a typed sub-graph defined in §5.

### 4.1 Sub-graph identity

Each sub-graph `G ∈ {I, D, E, V, C, Q, F, A}` is the pair `G = (N_G, E_G)` where:

- `N_G` is a finite set of typed nodes. Each node has a stable identifier `id` (locally unique within `G`) and a `type` drawn from the closed node-type set declared for `G` in §5.
- `E_G` is a finite set of typed edges. Each edge is `(source_id, target_id, predicate)` where `predicate` is drawn from the closed edge-predicate set declared for `G` in §5.

### 4.2 Cross-graph references

When an edge's `target_id` refers to a node in a different sub-graph, the target identifier MUST be prefixed with the target sub-graph's single-letter identifier. Specifically, target identifiers MUST match the regular expression

```
^(I|D|E|V|C|Q|F|A):[A-Za-z0-9_\-]+$
```

For example, a Claim node in C with a `supported_by` edge to a Figure node `fig_3a` in V MUST write the edge's `target_id` as `V:fig_3a`.

Local (intra-sub-graph) `target_id`s MUST NOT be prefixed.

### 4.3 Global ID uniqueness

Within a single package, after prefix-tagging, every node identifier (`<G.letter>:<id>`) MUST be unique. This enables a single global resolver from any cross-graph edge target. Implementations MUST refuse to emit a package that violates this rule.

### 4.4 Ablation semantics on the wire

When a `ECAA_ABLATE_*` flag (§7.4) is engaged at compose-time, the corresponding sidecar (or sidecar field) is absent — not present-but-empty. A package with any flag engaged is by definition non-conformant. Engaging multiple flags is permitted but each engages independently per the byte-identity contract in `operations.md` §4.4.

## 5. Sub-graphs

This section defines, for each of the eight sub-graphs: the node-type set, the edge-predicate set, cardinality requirements, materialization (sidecar path + line format), JSON Schema pointer, and OWL class pointer.

### 5.1 I — Intent

**Purpose.** Captures the SME's biological question, cohort definition, contrast specification, modality classification, and expected output schema.

**Node types** (closed, 5): `Question`, `Cohort`, `Contrast`, `Modality`, `ExpectedOutput`.

**Edge predicates** (closed, 3): `refines`, `stratifies`, `expects`.

**Cardinality requirements.**
- At least one `Question` node REQUIRED.
- Exactly one `Modality` node REQUIRED.
- At least one `ExpectedOutput` node REQUIRED.

**Materialization.** `runtime/intake-conversation.jsonl` — JSON Lines. Each line is a single typed node or edge serialization.

**JSON Schema.** `subgraph-schemas/intent.schema.json`.
**OWL classes.** `ecaa:IntentGraph` ⊇ {`ecaa:Question`, `ecaa:Cohort`, `ecaa:Contrast`, `ecaa:Modality`, `ecaa:ExpectedOutput`}.

### 5.2 D — Decision

**Purpose.** Captures method choices, justifications, alternatives considered, and supporting citations. Amendments to D produce lineage chains via `prov:wasDerivedFrom`.

**Node types** (closed, 4): `MethodChoice`, `Justification`, `Alternative`, `Citation`.

**Edge predicates** (closed, 5): `chooses`, `rejects`, `cites`, `amends`, `prov:wasDerivedFrom`.

**Cardinality requirements.**
- Each `MethodChoice` MUST have ≥1 outgoing `chooses` edge to a method identifier.
- Each `MethodChoice` MUST satisfy one of: ≥1 outgoing `cites` edge to a `Citation` node, OR `rationale` text field of length ≥ 30 characters. (This anchors Invariant 2 `decision_justification`.)
- A `MethodChoice` MAY have any number of `rejects` edges to `Alternative` nodes.

**Materialization.** `runtime/decisions.jsonl` — JSON Lines.

**JSON Schema.** `subgraph-schemas/decision.schema.json`.
**OWL classes.** `ecaa:DecisionGraph` ⊇ {`ecaa:MethodChoice`, `ecaa:Justification`, `ecaa:Alternative`, `ecaa:Citation`}.

### 5.3 E — Execution

**Purpose.** Captures workflow steps, container images, input/output bindings, and runtime environments.

**Node types** (closed, 5): `WorkflowStep`, `Container`, `InputFile`, `OutputFile`, `RuntimeEnvironment`.

**Edge predicates** (closed, 3): `produces`, `consumes`, `runs_in`.

**Cardinality requirements.**
- Each `WorkflowStep` MUST `runs_in` exactly one `RuntimeEnvironment`.
- Each `WorkflowStep` MUST `produces` at least one `OutputFile`.
- A `WorkflowStep` MAY `consumes` zero or more `InputFile`s.
- `Container` nodes SHOULD declare `image_ref` strings of the form `name@sha256:<digest>` (immutable digest). Mutable tags (`name:latest`) are PERMITTED but SHOULD be flagged.

**Materialization (two-materialization rule).** Sub-graph E is materialized in TWO places, both REQUIRED:

1. The WRROC `@graph` in `ro-crate-metadata.json` is the AUTHORITATIVE source for execution provenance (WfRun entities: `WorkflowStep`, `Container`, `InputFile`, `OutputFile`, `RuntimeEnvironment` correspond to `wfprov:` and `wfprov:`-aligned entities).
2. `runtime/validation-reports.jsonl` is an ECAA-typed shadow that captures per-task validation details (port-contract decisions, fan-out resolution indices, atom-validation outcomes) that the bare WRROC ontology does not carry.

Readers MUST consume the WRROC entities for the `WorkflowStep`/`Container`/`InputFile`/`OutputFile`/`RuntimeEnvironment` typed object and consult the JSONL only for ECAA-specific fields. Cross-consistency check: every `WorkflowStep` declared in the WRROC `@graph` MUST have a matching record in `validation-reports.jsonl`.

**JSON Schema.** `subgraph-schemas/execution.schema.json`.
**OWL classes.** `ecaa:ExecutionGraph` ⊇ {`ecaa:WorkflowStep`, `ecaa:Container`, `ecaa:InputFile`, `ecaa:OutputFile`, `ecaa:RuntimeEnvironment`}; each aligned to its `wfprov:` counterpart via `owl:equivalentClass` or `rdfs:subClassOf` in `ecaa-v0.1.ttl`.

### 5.4 V — Evidence

**Purpose.** Identifies the specific tables, figures, statistics, and files within the execution result that bear analytical weight.

**Node types** (closed, 4): `Table`, `Figure`, `Statistic`, `File`.

**Edge predicates** (closed, 2): `appears_in`, `computed_from`.

**Cardinality requirements.**
- Each `Statistic`, `Figure`, and `Table` MUST have ≥1 outgoing `computed_from` edge → an E `OutputFile`.
- An `appears_in` edge SHOULD point from a `Statistic`/`Figure`/`Table` to a document node (Markdown report, notebook) when applicable.

**Materialization.** `runtime/proofs.jsonl` — JSON Lines per port-contract proof. The RO-Crate `@graph` ALSO carries V entities as `dcat:Dataset` / `bioschemas:Dataset` / `schema:Image` types for ecosystem interoperability.

**JSON Schema.** `subgraph-schemas/evidence.schema.json`.
**OWL classes.** `ecaa:EvidenceGraph` ⊇ {`ecaa:Table`, `ecaa:Figure`, `ecaa:Statistic`, `ecaa:File`}; aligned to DCAT and bioschemas where applicable.

### 5.5 C — Claim

**Purpose.** Captures the narrative scientific claims an analysis makes, and links each claim to the Evidence that supports it.

**Node types** (closed, 3): `Claim`, `Quantification`, `Direction`.

**Edge predicates** (closed, 2): `supported_by`, `contradicts`.

**Status enum on `Claim`** (closed, 3 values): `verified`, `pending`, `contradicted`.

**Cardinality requirements.**
- Each `Claim` MUST satisfy one of: ≥1 outgoing `supported_by` edge to a V node (`Statistic`/`Figure`/`Table`), OR `status: "pending"`. (This anchors Invariant 1 `claim_completeness`.)
- A `Claim` with `status: "contradicted"` SHOULD have at least one `contradicts` edge to a V node.

**Materialization.** `runtime/claim-verification.json` — a single JSON document, not JSONL.

**JSON Schema.** `subgraph-schemas/claim.schema.json`.
**OWL classes.** `ecaa:ClaimGraph` ⊇ {`ecaa:Claim`, `ecaa:Quantification`, `ecaa:Direction`}.

### 5.6 Q — Equivalence

**Purpose.** Classifies re-execution outcomes per E `OutputFile` into one of a closed five-class set.

**Node types** (closed, 1): `RerunOutcome`.

**Class enum on `RerunOutcome.class`** (closed, 5): `byte_identical`, `semantic_equivalent`, `acknowledged_non_determinism`, `unavailable`, `failed`.

**Edge predicates** (closed, 2): `equivalent_to`, `diverges_from`.

**Cardinality requirements.**
- One `RerunOutcome` MUST be emitted per re-executed `OutputFile` in E.
- A `diverges_from` edge MUST carry per-modality tolerance metadata as JSON object on the edge (e.g., `{"tolerance": 1e-6, "metric": "L_inf"}`); the exact schema is implementation-specific.

**Materialization.** `runtime/verifier-decisions.jsonl` — JSON Lines.

**JSON Schema.** `subgraph-schemas/equivalence.schema.json`.
**OWL class.** `ecaa:RerunOutcome` ∈ `ecaa:EquivalenceGraph`; class enum encoded as `owl:oneOf` over named individuals.

### 5.7 F — Failure

**Purpose.** Represents blockers (typed recoverable failure states) and their recovery actions.

**Node types** (closed, 2): `Blocker`, `RecoveryAction`.

**Edge predicates** (closed, 2): `requires`, `unblocks`.

**`Blocker.kind` enum.** A closed 47-variant `BlockerKind` enum (see Appendix B for the full enumeration with one-line definitions). Implementations MUST emit only listed variants; extension is a minor-version bump.

**Cardinality requirements.**
- Every `Blocker` MUST satisfy one of: ≥1 outgoing `unblocks` edge to a `RecoveryAction`, OR a `resolved_at` timestamp recording the resolution time.
- A `RecoveryAction` MAY `requires` zero or more sibling `RecoveryAction`s (allowing recovery DAGs, not just chains).

**Materialization.** `runtime/assumptions.jsonl` — JSON Lines. (The filename retains its legacy "assumptions" name for backward compatibility with WRROC consumers; ECAA-v0.1-conformant readers MUST treat this file as the F sub-graph.)

**JSON Schema.** `subgraph-schemas/failure.schema.json`.
**OWL classes.** `ecaa:FailureGraph` ⊇ {`ecaa:Blocker`, `ecaa:RecoveryAction`}; `BlockerKind` enum encoded as `owl:oneOf` over the 47 named individuals.

### 5.8 A — Audit-proof

**Purpose.** Records machine-checkable invariant verdicts evaluated over the other seven sub-graphs.

**Node types** (closed, 1): `InvariantVerdict`.

**`InvariantVerdict.verdict` enum** (closed, 4): `Pass`, `Warn`, `Fail`, `Unverified`. See `invariants.md` §"Verdict ladder" for semantics.

**`InvariantVerdict.invariant_id` enum** (closed, 6): the six normative invariant IDs from §6.

**Edge predicates** (closed, 1): `evaluated_against`.

**Cardinality requirements.**
- Exactly 6 `InvariantVerdict` nodes MUST be present (one per invariant).
- Each `InvariantVerdict` MUST emit an `evaluated_against` edge to an IRI naming the package or specific sub-graph(s) evaluated.
- `evidence` field REQUIRED on every non-`Pass` verdict; OPTIONAL on `Pass`.

**Materialization.** `runtime/audit-proof-report.json` — a single JSON document. Shape normative; see `invariants.md` §"`audit-proof-report.json` shape".

**JSON Schema.** `subgraph-schemas/audit-proof.schema.json`.
**OWL class.** `ecaa:InvariantVerdict` ∈ `ecaa:AuditProofGraph`.

## 6. Audit-proof invariants

ECAA v0.1 defines six audit-proof invariants. The set is **closed** in v0.1. Full FOL predicates, verdict mapping, and reference-impl pointers are in `invariants.md`.

| # | ID | One-line | Default on violation | DL-expressible |
|---|---|---|---|---|
| 1 | `claim_completeness` | Every Claim has Evidence or `status: pending` | Warn | Yes (OWL DL) |
| 2 | `decision_justification` | Every MethodChoice has citations or rationale ≥30 chars | Warn | No (SHACL) |
| 3 | `evidence_coverage` | Every E OutputFile is referenced in V or marked unused in F | Warn | Yes (OWL DL) |
| 4 | `equivalence_failure` | Every Q `failed`/`acknowledged_non_determinism` outcome has a corresponding F Blocker | Fail | No (SHACL) |
| 5 | `cross_graph_integrity` | Every cross-graph reference dereferences | Fail | Yes (OWL DL) |
| 6 | `substrate_validity` | Package passes `runcrate validate ≥ 0.5.0` + 4 post-checks | Fail | No (external tool — SHACL stub only) |

### 6.1 Verdict ladder

Four normative verdicts: `Pass`, `Warn`, `Fail`, `Unverified`. Definitions and policy semantics in `invariants.md` §"Verdict ladder".

### 6.2 Determinism

Evaluators MUST be deterministic over package bytes. LLM-mediated predicates are NOT v0.1-conformant.

### 6.3 `audit-proof-report.json`

The report shape is normative — see `invariants.md` §"`audit-proof-report.json` shape". The shape declares `ecaa_version`, `package_iri`, `evaluated_at`, `min_reader_version`, an array of 6 `verdicts`, and an `evaluator` object.

### 6.4 Policy overrides

Implementations MAY override the default warn/fail mapping globally to warn-only mode (record as `evaluator.policy: "warn-only"`) or to strict mode (`evaluator.policy: "strict"` — every non-Pass becomes Fail). Per-invariant overrides are reserved for v0.2.

## 7. Validity-preserving operations

ECAA v0.1 defines four operations. The set is **closed** in v0.1. Full pre/post-condition contracts are in `operations.md`.

| Operation | Signature | Reference impl |
|---|---|---|
| `compose` | Intent → Package | `scripps-workflow intake` + `build` |
| `amend` | Package × DecisionDelta → Package | `scripps-workflow chat` `/amend` |
| `re-execute` | Package × Environment → Package | harness `--rerun` mode |
| `ablate` | Package × {AblationFlag} → NonConformantPackage | `ECAA_ABLATE_*` flags during compose |

### 7.1 `compose`

Takes an Intent sub-graph and produces a complete package P with all 8 sub-graphs materialized. Deterministic modulo documented exclusions (`evaluated_at` timestamp; `intake-conversation.jsonl` and `decisions.jsonl` content from SME interaction). See `operations.md` §1.

### 7.2 `amend`

Replaces or adds nodes in D. v0.1 limits amendments to D only; amendments to other sub-graphs are reserved for v0.2. Cross-graph-integrity (Invariant 5) MUST hold in the output package. See `operations.md` §2.

### 7.3 `re-execute`

Re-runs E on the same I + D against the current inputs; updates Q with class assignments; freshly evaluates A. See `operations.md` §3.

### 7.4 `ablate` — the load-bearing 6-flag contract

Six `ECAA_ABLATE_*` flags suppress exactly one sub-graph artifact each. The byte-identity contract (each flag suppresses ≥1 specific sidecar AND has no other observable effect on emission) is what makes the Aim 3A A vs. B′ design a mechanistic causal test of the ECAA layer rather than just a feature comparison. See `operations.md` §4 for the full table and contract guarantee.

## 8. Conformance

An implementation claims **ECAA v0.1 conformance** iff it passes ALL FIVE of the following bars on the D9 conformance suite (`crates/ecaa-conformance/`). There is no partial credit, no Core/Standard/Strict tiering, and no per-modality exemption.

### 8.1 Schema compliance

Every emitted package's eight sidecars MUST validate against their corresponding JSON Schemas under `subgraph-schemas/`. Validation tool: any JSON Schema draft-07-conformant validator.

### 8.2 Invariant evaluation

The implementation MUST evaluate the six audit-proof invariants and emit `runtime/audit-proof-report.json` matching the normative shape (`invariants.md` §"`audit-proof-report.json` shape"). Verdicts MUST be deterministic per §6.2.

### 8.3 Substrate round-trip

The package's `ro-crate-metadata.json` MUST round-trip through `runcrate validate ≥ 0.5.0` AND pass the four post-checks in §3.2.

### 8.4 Ablation contract

Each of the six `ECAA_ABLATE_*` flags MUST toggle exactly one sidecar (or sidecar field) per the table in §7.4 / `operations.md` §4.3. The D9 conformance suite verifies one-flag-at-a-time byte-identity of all other sidecars on a fixture corpus.

### 8.5 OWL projection

The implementation MUST produce a `package.ttl` RDF projection of the eight sub-graphs that passes OWL DL satisfiability against `ecaa-v0.1.ttl` AND SHACL conformance against `ecaa-v0.1.shacl.ttl`. Implementations MAY use the shipped `ecaa-v0.1.jsonld` JSON-LD context directly or produce equivalent RDF via their own emitter — D9 checks the resulting RDF, not the path.

### 8.6 No partial conformance

An implementation that passes four of five bars is NOT v0.1-conformant. Conformance is binary. Reports MUST cite the specific bar(s) failed.

### 8.7 Evaluator licensing

The conformance suite does NOT require open-source evaluators. Any implementation passing all five bars MAY claim conformance, regardless of license.

## 9. Versioning and version negotiation

ECAA follows Semantic Versioning ([semver.org](https://semver.org/)) — `MAJOR.MINOR.PATCH`. This document is v0.1.0.

### 9.1 Change classification

- **MAJOR.** Breaking changes to sub-graph schemas, removed/renamed node types or edge predicates, removed/renamed invariants, removed/renamed verdicts, removed/renamed ablation flags, or any change that would invalidate a previously-conformant package.
- **MINOR.** Additive changes: new optional fields, new (optional-from-the-impl-side) invariants, new closed-set members (BlockerKind variants, Q-classes — provided readers MUST tolerate unknowns per §9.4), new operations (provided they preserve existing contracts), new substrate bindings.
- **PATCH.** Editorial fixes (spelling, formatting); machine-readable companion fixes that do not change normative content.

### 9.2 Per-package version declarations

Every ECAA package MUST declare its specification version in `audit-proof-report.json` at the top level:

```json
{
  "ecaa_version": "0.1",
  "min_reader_version": "0.1",
  "max_reader_version": null,
  ...
}
```

- `ecaa_version` (REQUIRED) — the specification version this package was emitted against. SemVer string.
- `min_reader_version` (REQUIRED) — the minimum reader version that can correctly interpret this package. SemVer string.
- `max_reader_version` (OPTIONAL) — the maximum reader version that should attempt to interpret this package. `null` means no upper bound.

### 9.3 Reader-side declarations

Every ECAA-v0.1-conformant reader MUST publish its `reader_supports` range — a SemVer range expression (e.g., `"0.1.x"` or `">=0.1, <0.3"`) — in its documentation or version-introspection API. A reader's effective support range is the intersection of its declared range and the v0.x range of this specification.

### 9.4 Compatibility rule

A reader R is **compatible** with a package P iff:

```
R.reader_version ∈ [P.min_reader_version, P.max_reader_version ?? ∞]
```

On incompatibility, R MUST refuse to process P and MUST surface a typed error. Reference implementations route this to `BlockerKind::UnsupportedECAAVersion`.

**Intra-major compatibility (SemVer guarantee).** Within the same major version (e.g., v0.x), readers MUST be backward-compatible with packages of any minor or patch version with `min_reader_version ≤ reader_version`. Producers MUST NOT raise `min_reader_version` for purely additive changes.

**Inter-major compatibility.** No compatibility guarantee unless the producer explicitly declares it via `compat_artifacts` (§9.5).

### 9.5 Forward-compatibility — skip-and-continue rule

Readers MUST tolerate unknown files in `runtime/` and unknown fields in JSONL line objects (skip-and-continue), provided the unknown fields are not under a `required: true` JSON Schema constraint. This is the minimum forward-compatibility guarantee.

### 9.6 `compat_artifacts` (optional)

A package MAY ship alternate-version artifacts under `runtime/compat/`. Implementations MAY declare these in `audit-proof-report.json`:

```json
{
  "compat_artifacts": [
    {"version": "0.2", "path": "runtime/compat/audit-proof-report-v0.2.json"}
  ]
}
```

A v0.2 reader encountering a v0.1 package with a `compat_artifacts` entry for v0.2 MAY use the alternate artifact instead of the v0.1 original. This is OPTIONAL; producers are NOT required to ship compat artifacts.

## 10. Security considerations

ECAA packages may contain pointers to executable code (agent-generated) and references to clinical data. The specification addresses four security surfaces.

### 10.1 Execution trust

ECAA packages declare WHAT was executed; they do NOT declare that the execution was SAFE. Re-executing a package MUST be done in an isolated sandbox. Implementations SHOULD declare a sandbox profile (e.g., `bubblewrap`, `apptainer --containall`, AWS isolated VPC) in the `RuntimeEnvironment` node's `sandbox_profile` field. Readers SHOULD refuse re-execution when no sandbox profile is declared, unless the operator explicitly opts in.

### 10.2 Supply-chain attestation

Container references in E `Container` nodes SHOULD use immutable digests (SHA-256) rather than mutable tags. The recommended `image_ref` format is `name@sha256:<digest>`. ECAA-v0.1-conformant emitters SHOULD warn when emitting mutable-tag references. Future minor versions MAY require digest references for selected modalities.

### 10.3 Package signing

ECAA v0.1 does NOT mandate package signing. Implementations MAY sign `audit-proof-report.json` out-of-band using Sigstore or any S/MIME-equivalent system. Signed-manifest profiles are reserved for v0.2.

### 10.4 PII / PHI

ECAA packages MAY reference clinical or otherwise restricted data. Direct PII or PHI MUST NOT appear in any sub-graph node. The Evidence sub-graph's `File` nodes MUST carry an `access_control` annotation when their content is restricted; readers MUST refuse to publish artifacts whose `access_control` is anything other than `public` without explicit operator authorization.

### 10.5 Reproducibility ≠ trust

A v0.1-conformant package establishes machine-checkable consistency between its claims, evidence, decisions, and execution provenance. It does NOT establish biological validity. Readers and reviewers MUST treat conformance verdicts as a hygiene floor, not a quality ceiling.
